# MID-89 Smoke Hang Postmortem

## Root cause

The failing main-branch smoke runs are best explained by a race in the smoke harness' initial socket setup, not by `host:create` failing to ack. In `scripts/smoke.ts`, `main()` opens four Socket.IO clients immediately (`host`, `display`, `a`, `b`) at lines 49-52, but only starts awaiting their `connect` events later and sequentially at lines 69-72. Because `display`, `a`, or `b` can connect before its corresponding `await new Promise(... s.on("connect" ...))` is installed, the promise can wait forever on an event that already fired. That produces exactly the observed CI shape: zero smoke output, no `pin:`, and a hard timeout. The absence of `pin:` does not prove the script is stuck at line 74; it is also consistent with being stuck on lines 70-72 before `host.emit("host:create", quiz, r)` is ever executed. By contrast, the current `server.ts` `host:create` handler registers inside `io.on("connection")` and, once invoked, calls `createGame`, joins the room, and acks synchronously at lines 65-80. In cluster C, the same handler was wrapped in validation/try-catch and still acks valid quizzes with `{ ok: true, pin }`; there is no file:line evidence that a valid `host:create` can be received and then silently fail to ack.

## Why cluster A and cluster C tripped it, but the revert did not

The source history points to nondeterministic timing rather than a deterministic regression. Cluster A (`f48d0af`) failed main smoke, cluster C (`ae2c419`) failed main smoke, and the revert (`6ba282a`) passed with the same 12-scenario smoke shape as post-cluster-A main. The failed logs show `npx wait-on http://localhost:4321 --timeout 60000` warming `/` through a burst of `HEAD / 200` requests after `▶ broadcast ready`, then smoke runs and emits nothing until `[smoke] hard timeout 180s`. That readiness pattern makes the dev server hot enough that multiple websocket clients can connect very quickly once `scripts/smoke.ts` starts. Sometimes the sequential awaits catch those events; sometimes one client has already connected before its listener exists. Cluster A and C likely shifted runtime timing enough to expose the race; the revert run merely landed on the passing side of the same race. PR CI did not catch this because `.github/workflows/pr.yml` stops after typecheck and coverage, while `.github/workflows/main.yml` is the only workflow that boots the dev server and runs smoke.

## Cluster C assessment

Cluster C should not stay reverted because of this smoke hang. Its validator changes do alter the `host:create` ack shape from `{ pin }` to `{ ok: true, pin }`, but the existing smoke helper destructures `pin` and the added cluster C scenarios explicitly expected the widened shape. The valid-quiz path validates synchronously, creates the game, attaches the host, joins the socket room, acks, and broadcasts; the invalid path catches and acks `{ ok: false, reason: "invalid-quiz" }`. I did not find evidence that F5 or F10 introduced an independent handler bug that would explain a first-create hang. CSV neutralization is also isolated to export formatting.

## Minimum-risk fix

The reliable fix is to remove the smoke harness race before treating any main-only smoke failure as product signal. Change the initial `main()` setup to use the existing safe pattern from `connectSock()` or equivalent: install each socket's `connect` listener at creation time and either create the four sockets via `await Promise.all([connectSock(), connectSock(), connectSock(), connectSock()])`, or make a `waitConnected(socket)` helper that resolves immediately when `socket.connected` is already true and otherwise waits with `once("connect")` plus a short timeout. This is a test-harness-only change and does not affect runtime behavior.

For CI, the minimum workflow adjustment is diagnostic/stabilizing rather than a substitute for the harness fix: keep the dev-server boot, but change readiness to avoid Next's `/` compile burst and add an explicit socket-capable warmup/check before smoke. For example, wait on `tcp:4321` or a cheap health endpoint instead of `http://localhost:4321`, then run smoke with bounded per-ack/per-connect timeouts so the log reports the exact stuck phase. A `curl /` warmup may reduce Next compile noise, but it does not fix the missed-`connect` race by itself. After the harness fix, add the same smoke job to PR CI or a required PR workflow so cluster-level changes fail before merge instead of only on `main`.
