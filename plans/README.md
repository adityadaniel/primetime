## Lower-priority notes (real, but not turned into standalone plans)

Captured so they aren't re-audited from scratch. Promote to a plan if they start
to bite.

- **In-memory rate limiter (`lib/rate-limit.ts`, `MID-79` TODO):** by-design for
  the current single-process deployment; only incorrect under multi-instance
  horizontal scaling. Needs a shared store (Redis) — revisit if/when the app is
  load-balanced.
- **`PERF-02` — 25+ `useState` in the Q&A control client** (`control-client.tsx`):
  the question board is already memoized, so the practical render cost is smaller
  than the raw hook count suggests. Group related state into reducers only if
  profiling shows real jank.
- **`PERF-03` — word-cloud snapshot re-sort** (`lib/wordcloud.ts` `snapshotWords`):
  O(N log N) per broadcast; profile with realistic word/player counts before
  adding a snapshot cache.
- **`SEC-03` — Apify token passed as a URL query param** (`lib/wonderwall-apify.ts:103`):
  server→Apify call; prefer an `Authorization` header to keep the token out of
  logs. Low severity; bundle into any future WonderWall-analysis work.
- **`SEC-07/08/09` — defense-in-depth:** add a CSP on the public display/upload
  surfaces; confirm SameSite posture and consider explicit CSRF tokens on
  state-changing API routes (Next server actions already have CSRF built in);
  redact secrets from third-party error logging. None are open holes today.
- **`DEPS-01` — Prisma 6→7 major lag:** real, but L effort and not urgent; do as a
  dedicated migration plan when there's appetite.
- **`DEPS-02` — `next-auth` on `5.0.0-beta.31`:** pin and monitor for GA; do **not**
  downgrade to v4 (v5 is the correct line for the App Router — the audit
  subagent's downgrade suggestion was rejected).
- **`DX-02` — stale `PLAYER_CAP` env var:** `lib/config.ts:133` uses the code-level
  constant `PLAYER_CAP`; the env var is inert. Remove it from `.env.example` /
  add a clarifying comment (tiny doc cleanup).
- **`CORRECTNESS-5` — clock-skew scoring** (`lib/game.ts` score calc): already
  clamped to `[0,1]`; theoretical. Add a `msFromStart < 0` warning log if desired.

## Findings considered and rejected

So nobody re-audits them:

- **Committed secrets (audit SEC-01/02/10):** REJECTED — `.env` and `.env.local`
  are gitignored (`.gitignore:3-4`) and never committed (`git log --all -- .env`
  is empty). These are local-only dev files, not a repo exposure. The audit did
  encounter a local Apify API token in transient agent output; no value is
  recorded here. Rotate that token if the output was persisted or shared.
- **Image-upload not wired (audit DIRECTION-03):** REJECTED — fully implemented in
  the quiz builder `app/host/quiz/new/page.tsx` (uploads via `/api/upload`,
  `patch('imageUrl', …)`, "ADD STILL/REPLACE" control, preview) per MID-278. The
  subagent inspected a stale path (`app/host/page.tsx`).
- **Missing auth rate-limiting (audit SEC-11):** REJECTED — signup is already
  rate-limited (see `lib/rate-limit.ts` usage + `lib/auth-flows` tests).
- **Map delete-during-iteration (audit CORRECTNESS-3):** REJECTED — deleting the
  current/visited key during `Map` iteration is well-defined and safe in JS; the
  subagent conceded it "works in practice." Not a bug.
- **Repository base-class extraction (audit ARCH-04):** REJECTED as a plan — L
  effort, HIGH refactor risk, MED confidence, speculative ROI. Revisit only if a
  new session-type feature needs to be built three times in lockstep.
- **WonderWall polling vs Socket.IO (audit ARCH-02):** NOT A FINDING — the 8s
  polling cadence is an intentional design choice (`MID-405`), acceptable for the
  moderation-queue UX. Revisit only if latency/scale becomes a real complaint.
